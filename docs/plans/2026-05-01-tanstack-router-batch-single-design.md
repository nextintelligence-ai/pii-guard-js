# TanStack Router 기반 단일/일괄 처리 화면 분리 설계

- 작성일: 2026-05-01
- 상태: 사용자 방향 승인 후 설계 고정
- 범위: 라우팅 구조, 홈 화면, 단일 처리 화면, 여러 PDF 일괄 자동 처리 화면, 파일별 검수 상세 화면

## 1. 배경

현재 앱은 `src/App.tsx` 하나가 단일 PDF 처리 흐름을 모두 담당한다.

- PDF 1개 열기
- 정규식/OCR/NER 후보 탐지
- 후보 검수와 수동 마스크 편집
- MuPDF redaction 적용
- post-check 누수 검증
- 결과 PDF 다운로드

새 요구사항은 TanStack Router를 사용해 화면을 분리하고, 단일 PDF 처리와 여러 PDF 일괄 처리를 별도 작업 모드로 제공하는 것이다.

## 2. 사용자 확정 결정

1. 홈(`/`)은 단일/일괄 처리 중 하나를 고르는 작업 시작 화면으로 둔다.
2. batch는 여러 PDF 파일을 한 번에 처리하는 의미다.
3. batch 기본 동작은 자동 적용 중심이다.
4. batch는 여러 파일을 넣으면 자동 탐지, 자동 익명화 적용, 검증, 결과 확인까지 진행한다.

## 3. 핵심 원칙

1. PDF 원본은 외부 네트워크로 나가지 않는다.
2. PWA/Service Worker는 추가하지 않는다.
3. 여러 PDF를 동시에 워커에 올리지 않는다. batch는 메모리 부담을 줄이기 위해 한 번에 한 파일씩 순차 처리한다.
4. 단일 처리 화면의 기존 검수/편집 경험은 유지한다.
5. batch 자동 적용 기본 범위는 정규식 후보와 OCR 정규식 후보로 제한한다.
6. NER 후보는 기존 설계처럼 기본 OFF다. batch에서 NER 자동 적용은 명시 옵션으로만 켠다.
7. post-check 누수 검증 결과가 0건이 아닌 파일은 성공으로 보지 않고 검증 경고 상태로 둔다.

## 4. 라우팅 구조

TanStack Router는 작은 라우트 수와 기존 구조를 고려해 code-based route tree로 시작한다. file-based 라우팅과 generated `routeTree.gen.ts`는 라우트가 늘어나거나 파일 기반 컨벤션이 필요해질 때 도입한다.

예상 경로:

| 경로 | 역할 |
| --- | --- |
| `/` | 홈. 단일/일괄 처리 시작 선택 |
| `/single` | 단일 PDF 처리. 현재 `App.tsx` 경험을 이동 |
| `/batch` | 여러 PDF 일괄 자동 처리 큐 |
| `/batch/$jobId` | batch 안 특정 파일의 검수/재처리 상세 |

공통 root route는 앱 셸을 렌더링한다.

- 상단 내비게이션: 단일 처리, 일괄 처리, 도움말
- route outlet
- 전역 toast
- 사용법 모달

## 5. 홈 화면

홈은 마케팅 랜딩이 아니라 작업 시작 화면이다. 개인정보 도구 특성상 최근 파일명, 처리 이력, 로컬 히스토리는 기본 표시하지 않는다.

```text
/
┌──────────────────────────────────────────────────────────────┐
│ PDF 익명화 도구        [단일 처리] [일괄 처리]      [도움말]  │
├──────────────────────────────┬───────────────────────────────┤
│ 단일 PDF 처리                 │ 여러 PDF 자동 처리              │
│                              │                               │
│ [PDF 1개 선택/드롭]           │ [PDF 여러 개 선택/드롭]          │
│                              │                               │
│ 검수 후 직접 적용             │ 자동 탐지 → 자동 적용 → 결과 확인 │
└──────────────────────────────┴───────────────────────────────┘
```

동작:

- 왼쪽에 PDF 1개를 선택하거나 드롭하면 `/single`로 이동하고 해당 파일을 연다.
- 오른쪽에 PDF 여러 개를 선택하거나 드롭하면 `/batch`로 이동하고 큐에 추가한다.
- 파일 없이 상단 내비게이션을 눌러도 각 작업 화면으로 이동할 수 있다.

## 6. 단일 처리 화면

현재 앱의 기본 경험을 유지한다. 기존 `App.tsx`는 route component 또는 `SinglePage`로 분리한다.

```text
/single
┌──────────────────────────────────────────────────────────────┐
│ PDF 익명화 도구   [단일 처리] [일괄 처리]        [도움말]     │
├────────────────┬─────────────────────────────────────────────┤
│ 후보 검수       │ PDF 미리보기 / 박스 편집                    │
│ - 정규식 후보   │ - 페이지 이동                               │
│ - OCR 후보      │ - 수동 마스크                               │
│ - NER 후보      │                                             │
│ - 직접 마스크   │                                             │
└────────────────┴─────────────────────────────────────────────┘
```

단일 처리의 상태는 기존 Zustand store와 hooks를 최대한 재사용한다.

- `usePdfDocument`
- `useAutoDetect`
- `useOcrDetect`
- `useApply`
- `useKeyboard`
- `CandidatePanel`
- `PdfCanvas`
- `PageNavigator`

## 7. Batch 큐 화면

batch는 여러 PDF를 추가하고, 사용자가 처리 시작을 누르면 순차 자동 처리한다.

```text
/batch
┌──────────────────────────────────────────────────────────────┐
│ PDF 익명화 도구   [단일 처리] [일괄 처리]        [도움말]     │
├──────────────────────────────────────────────────────────────┤
│ [PDF 추가] [처리 시작] [일시정지] [성공 파일 저장] [목록 비우기] │
├──────────────────────────────────────────────────────────────┤
│ 설정: ☑ OCR 사용   ☐ NER 후보도 자동 적용                    │
├──────────────────────────────────────────────────────────────┤
│ 요약: 전체 12 · 완료 8 · 처리 중 1 · 실패 2 · 검증 경고 1     │
├──────────────────────────────────────────────────────────────┤
│ 파일명             상태          후보   적용   검증   작업     │
│ claim-a.pdf        완료          14     완료   0건    [저장]   │
│ claim-b.pdf        처리 중        ...    ...    ...    [중단]   │
│ claim-c.pdf        검증 경고      9      완료   1건    [검수]   │
│ claim-d.pdf        암호 필요      -      -      -      [재시도] │
└──────────────────────────────────────────────────────────────┘
```

### 7.1 Batch 기본 처리 흐름

```text
여러 PDF 추가
  → 처리 시작
  → 다음 대기 파일 선택
  → MuPDF worker open
  → 정규식 자동 탐지
  → OCR 사용 설정이면 필요한 페이지 OCR
  → 자동 적용 대상 box 구성
  → MuPDF redaction apply
  → post-check 누수 검증
  → 성공/검증 경고/실패 상태 기록
  → worker 문서 close 또는 reset
  → 다음 파일 처리
```

### 7.2 자동 적용 정책

기본 자동 적용:

- 정규식 후보: 주민등록번호, 전화번호, 이메일, 계좌번호, 사업자번호, 카드번호, 주소
- OCR 정규식 후보: OCR line에서 기존 정규식 detector로 찾은 후보

기본 제외:

- NER 후보: 사람 이름, 비정형 주소, URL, 날짜, 시크릿
- OCR NER 후보

옵션:

- `NER 후보도 자동 적용`을 사용자가 켜면 NER threshold 이상 후보만 자동 적용한다.
- 이 옵션은 batch 설정에 명확히 표시하고 기본값은 OFF다.

## 8. Batch 상세 검수 화면

검증 경고, 암호 재시도 후 검수, 특정 파일 수동 보강이 필요한 경우 `/batch/$jobId`로 들어간다.

```text
/batch/$jobId
┌──────────────────────────────────────────────────────────────┐
│ ← 일괄 목록   claim-c.pdf   검증 경고 1건      [다시 적용]    │
├────────────────┬─────────────────────────────────────────────┤
│ 후보/마스크     │ PDF 미리보기 / 박스 편집                    │
│ 검증 경고 표시  │ 기존 단일 처리 편집 경험 재사용             │
└────────────────┴─────────────────────────────────────────────┘
```

상세 화면은 단일 처리 편집 UI를 재사용한다. 차이는 batch job context를 표시하고, 적용 결과를 batch 큐의 해당 job에 다시 반영한다는 점이다.

## 9. Batch 상태 모델 초안

batch는 기존 단일 store와 분리된 batch store를 둔다. 기존 store는 “현재 열려 있는 PDF 편집 세션”을 담당하고, batch store는 파일 큐와 결과 메타데이터를 담당한다.

```ts
type BatchJobStatus =
  | 'queued'
  | 'opening'
  | 'detecting'
  | 'ocr'
  | 'applying'
  | 'done'
  | 'warning'
  | 'failed'
  | 'cancelled';
```

job이 보관할 정보:

- `id`
- `file`
- `fileName`
- `status`
- `candidateCount`
- `enabledBoxCount`
- `report`
- `outputBlob`
- `errorMessage`
- `needsReview`

주의:

- batch store에 PDF binary를 오래 보관하지 않는다. 사용자가 선택한 `File` 객체와 처리 결과 Blob만 보관한다.
- 현재 MuPDF worker가 단일 문서 중심 API이므로 batch runner는 파일을 하나씩 열고 처리한다.
- batch 상세 검수로 들어갈 때만 해당 job 파일을 단일 편집 store에 로드한다.

## 10. 오류와 경고 상태

| 상황 | 상태 | 사용자 액션 |
| --- | --- | --- |
| 비밀번호 필요 | `failed` 또는 `warning` | 비밀번호 입력 후 재시도 |
| PDF open 실패 | `failed` | 제거 또는 재시도 |
| OCR 실패 | `warning` | OCR 없이 계속, 또는 상세 검수 |
| 적용 실패 | `failed` | 상세 확인 후 재시도 |
| post-check leaks > 0 | `warning` | 상세 검수에서 누락 박스 보강 후 다시 적용 |
| 사용자가 중단 | `cancelled` | 재시작 가능 |

## 12. 테스트 전략

1. 라우팅 smoke 테스트
   - `/`가 홈을 렌더링한다.
   - `/single`이 단일 처리 화면을 렌더링한다.
   - `/batch`가 batch 큐 화면을 렌더링한다.

2. batch store 단위 테스트
   - 여러 파일 추가
   - 상태 전이
   - 성공/실패/경고 카운트 요약
   - 목록 비우기와 취소

3. batch runner 단위/통합 테스트
   - mock worker로 queued → detecting → applying → done 전이 확인
   - post-check leaks > 0이면 warning 처리
   - 한 파일 실패 후 다음 파일이 계속 처리되는지 확인

4. 기존 redaction 통합 테스트 유지
   - batch 자동 적용도 최종적으로 기존 `apply` 경로의 누수 검증을 우회하지 않는다.

## 14. 미해결 결정

1. 성공 파일 전체 저장 방식
   - 브라우저에서 ZIP을 만들려면 새 의존성이 필요하다.
   - 1차 구현은 성공 파일을 개별 저장하거나 순차 다운로드로 시작하는 것이 보수적이다.

2. 암호 PDF batch UX
   - 처리 중 prompt를 띄울지, 암호 필요 상태로 남기고 사용자가 재시도하게 할지 결정해야 한다.
   - batch 자동 처리에서는 prompt로 전체 큐를 막지 않는 편이 좋다.

3. batch 중 OCR 실패 정책
   - OCR 실패 파일을 warning으로 남기고 정규식 후보만 적용할지, OCR 실패를 전체 실패로 볼지 결정해야 한다.
   - 1차 추천은 warning이다.
